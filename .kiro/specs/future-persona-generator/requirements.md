# Requirements Document

## Introduction

The Future Persona Generator is a module within the Future Self Voice Simulator project. It constructs a psychologically coherent "future self" persona from user-supplied inputs — including current state, life goals, personality traits, and scenario type — by prompting a large language model (GPT-4o or Claude). The module returns a structured `Persona` object containing a narrative summary, an emotional tone, a key message, and a life outcome. The persona is projected 5–10 years into the future and must be realistic, grounded, and free of contradictions. It exposes a REST API via FastAPI and a React frontend component (`PersonaBuilder`). The generated `Persona` is consumed downstream by the Scenario Engine and Dialogue Generator modules.

---

## Glossary

- **PersonaGenerator_API**: The FastAPI application that handles persona generation and retrieval requests.
- **LLM_Client**: The internal service layer that constructs prompts and communicates with a configured large language model provider (GPT-4o or Claude).
- **LLM_Provider**: An external large language model service — either OpenAI (GPT-4o) or Anthropic (Claude).
- **UserProfile**: The input data model containing `current_age`, `years_ahead`, `goals`, `current_state`, `personality_traits`, and `scenario_type` submitted by the user.
- **Persona**: The output data model containing `persona_id`, `summary`, `tone`, `key_life_events`, `life_outcome`, and `key_message` produced by the LLM_Client.
- **persona_id**: A unique string identifier assigned to a generated Persona at creation time.
- **Scenario_Type**: A user-declared outcome framing — one of `"success"`, `"regret"`, or `"neutral"` — that determines the tone and life outcome of the generated Persona.
- **Tone**: The narrative register of the generated Persona — derived deterministically from `scenario_type`: `"success"` → `"confident, fulfilled"`, `"regret"` → `"reflective, slightly heavy"`, `"neutral"` → `"calm, observational"`.
- **Life_Outcome**: A concise statement describing the future self's overall life situation, grounded in the user's goals and scenario type.
- **Key_Message**: The single most important thing the future self wants to communicate to the present self.
- **Key_Life_Events**: An ordered list of plausible future milestones included in the Persona, generated by the LLM_Client. Must be realistic within a 5–10 year horizon.
- **PersonaBuilder**: The React frontend component that collects UserProfile inputs, submits them to the PersonaGenerator_API, and renders the resulting Persona.
- **Scenario_Engine**: The downstream module that consumes a Persona to generate situational scenarios for the user's future self.
- **Dialogue_Generator**: The downstream module that consumes a Persona to produce voice-ready dialogue lines.
- **Persona_Store**: The persistence layer that stores and retrieves Persona records by `persona_id`.

## Requirements

### Requirement 1: Persona Generation via LLM

**User Story:** As a user, I want to submit my current state, goals, personality traits, and scenario type, so that the system generates a coherent narrative description of my future self 5–10 years from now.

#### Acceptance Criteria

1. WHEN a POST request is made to `/persona/generate` with a valid UserProfile, THE PersonaGenerator_API SHALL forward the UserProfile to the LLM_Client and return HTTP 201 with the generated Persona as a JSON object.
2. THE LLM_Client SHALL construct a prompt that incorporates `current_age`, `years_ahead`, `goals`, `current_state`, `personality_traits`, and `scenario_type` from the UserProfile.
3. WHEN the LLM_Provider returns a response, THE LLM_Client SHALL parse it into a Persona containing `persona_id`, `summary`, `tone`, `key_life_events`, `life_outcome`, and `key_message`.
4. THE PersonaGenerator_API SHALL assign a unique `persona_id` to each generated Persona at the time of creation.
5. WHEN two generation requests are processed concurrently, THE PersonaGenerator_API SHALL assign distinct `persona_id` values to each resulting Persona.
6. IF the LLM_Provider returns an error or times out, THEN THE PersonaGenerator_API SHALL return HTTP 502 with a descriptive error message and SHALL NOT persist a Persona.

---

### Requirement 2: UserProfile Input Validation

**User Story:** As a developer, I want all UserProfile inputs to be validated before processing, so that the LLM_Client receives only well-formed data.

#### Acceptance Criteria

1. WHEN a POST request to `/persona/generate` is received, THE PersonaGenerator_API SHALL validate that `current_age` is an integer between 18 and 100 inclusive.
2. WHEN a POST request to `/persona/generate` is received, THE PersonaGenerator_API SHALL validate that `years_ahead` is an integer between 5 and 10 inclusive, reflecting the 5–10 year projection constraint.
3. WHEN a POST request to `/persona/generate` is received, THE PersonaGenerator_API SHALL validate that `goals` is a non-empty list containing between 1 and 10 strings, each no longer than 200 characters.
4. WHEN a POST request to `/persona/generate` is received, THE PersonaGenerator_API SHALL validate that `current_state` is a non-empty string no longer than 500 characters describing the user's present life situation.
5. WHEN a POST request to `/persona/generate` is received, THE PersonaGenerator_API SHALL validate that `personality_traits` is a non-empty list containing between 1 and 5 strings, each no longer than 50 characters.
6. WHEN a POST request to `/persona/generate` is received, THE PersonaGenerator_API SHALL validate that `scenario_type` is one of the values `"success"`, `"regret"`, or `"neutral"`.
7. WHEN any UserProfile field fails validation, THE PersonaGenerator_API SHALL return HTTP 422 with a JSON body identifying each invalid field and a descriptive error message per field.
8. WHEN all UserProfile fields pass validation, THE PersonaGenerator_API SHALL proceed with persona generation without modification to the input values.

---

### Requirement 3: Scenario Type Tone Alignment

**User Story:** As a user, I want the generated persona's tone to match my chosen scenario type, so that the future self description feels psychologically authentic and consistent.

#### Acceptance Criteria

1. THE LLM_Client SHALL include the `scenario_type` value in the prompt in a way that instructs the LLM_Provider to adopt the corresponding narrative register.
2. WHEN `scenario_type` is `"success"`, THE LLM_Client SHALL instruct the LLM_Provider to produce a Persona with `tone` set to `"confident, fulfilled"` and a `life_outcome` reflecting achieved major goals.
3. WHEN `scenario_type` is `"regret"`, THE LLM_Client SHALL instruct the LLM_Provider to produce a Persona with `tone` set to `"reflective, slightly heavy"` and a `life_outcome` reflecting failure or abandonment of key goals.
4. WHEN `scenario_type` is `"neutral"`, THE LLM_Client SHALL instruct the LLM_Provider to produce a Persona with `tone` set to `"calm, observational"` and a `life_outcome` reflecting a mixed or average outcome.
5. THE PersonaGenerator_API SHALL reject any Persona returned by the LLM_Provider whose `tone` value does not correspond to the requested `scenario_type`, and SHALL retry the LLM_Provider request up to 2 additional times before returning HTTP 502.
6. THE LLM_Client SHALL include all values from the `goals` list in the prompt so that the generated `summary`, `life_outcome`, and `key_life_events` reflect the user's stated intentions.
7. THE LLM_Client SHALL instruct the LLM_Provider that the Persona must contain no internal contradictions between `life_outcome`, `tone`, and `key_life_events`.

---

### Requirement 4: Persona Structure and Content Completeness

**User Story:** As a developer, I want every generated Persona to contain all required fields with valid values, so that downstream modules can consume it without additional transformation.

#### Acceptance Criteria

1. THE PersonaGenerator_API SHALL return a Persona JSON object containing exactly the fields `persona_id`, `summary`, `tone`, `key_life_events`, `life_outcome`, and `key_message` in every successful generation response.
2. THE LLM_Client SHALL instruct the LLM_Provider to produce a `summary` that is a prose narrative of no fewer than 100 words and no more than 500 words.
3. THE LLM_Client SHALL instruct the LLM_Provider to produce a `key_life_events` list containing between 3 and 7 items, each a non-empty string no longer than 150 characters, all plausible within a 5–10 year horizon.
4. THE LLM_Client SHALL instruct the LLM_Provider to produce a `life_outcome` that is a non-empty string no longer than 200 characters summarising the future self's overall life situation.
5. THE LLM_Client SHALL instruct the LLM_Provider to produce a `key_message` that is a non-empty string no longer than 150 characters representing the single most important thing the future self wants to say.
6. WHEN the LLM_Provider returns a `summary` outside the 100–500 word range, THE PersonaGenerator_API SHALL log a warning and retry the LLM_Provider request up to 2 additional times before returning HTTP 502.
7. WHEN the LLM_Provider returns a `key_life_events` list with fewer than 3 or more than 7 items, THE PersonaGenerator_API SHALL log a warning and retry the LLM_Provider request up to 2 additional times before returning HTTP 502.
8. THE PersonaGenerator_API SHALL NOT expose raw LLM_Provider response text to the client; it SHALL return only the structured Persona fields.

---

### Requirement 5: Persona Persistence

**User Story:** As a developer, I want generated personas to be stored after creation, so that they can be retrieved later by the user and by downstream modules.

#### Acceptance Criteria

1. WHEN the LLM_Client successfully generates a Persona, THE PersonaGenerator_API SHALL persist the Persona record in the Persona_Store before returning the HTTP 201 response.
2. THE PersonaGenerator_API SHALL store `persona_id`, `summary`, `tone`, `key_life_events`, `life_outcome`, `key_message`, and a `created_at` timestamp in UTC ISO 8601 format for each persisted Persona.
3. WHEN a Persona is persisted, THE PersonaGenerator_API SHALL store the originating UserProfile fields (`current_age`, `years_ahead`, `goals`, `current_state`, `personality_traits`, `scenario_type`) alongside the Persona record.
4. IF the Persona_Store is unavailable at the time of write, THEN THE PersonaGenerator_API SHALL return HTTP 503 with a descriptive error message and SHALL NOT return a `persona_id` to the client.

---

### Requirement 6: Persona Retrieval

**User Story:** As a developer, I want to retrieve a previously generated persona by its ID, so that downstream modules can load it on demand.

#### Acceptance Criteria

1. WHEN a GET request is made to `/persona/{id}` with a valid `persona_id`, THE PersonaGenerator_API SHALL return HTTP 200 with the corresponding Persona as a JSON object.
2. WHEN a GET request is made to `/persona/{id}` with a `persona_id` that does not exist in the Persona_Store, THE PersonaGenerator_API SHALL return HTTP 404 with a descriptive error message.
3. THE PersonaGenerator_API SHALL return the Persona JSON with fields `persona_id`, `summary`, `tone`, `key_life_events`, `life_outcome`, and `key_message` in every successful retrieval response.
4. WHEN a Persona is retrieved, THE PersonaGenerator_API SHALL return the identical field values that were stored at creation time, without modification.

---

### Requirement 7: LLM Provider Abstraction

**User Story:** As a developer, I want the system to support multiple LLM providers, so that the generation backend can be swapped without changing the API contract.

#### Acceptance Criteria

1. THE LLM_Client SHALL support GPT-4o (OpenAI) and Claude (Anthropic) as interchangeable LLM_Provider backends.
2. WHEN a provider is configured, THE LLM_Client SHALL route all generation requests to that provider exclusively.
3. WHERE GPT-4o is configured as the LLM_Provider, THE LLM_Client SHALL use the OpenAI Chat Completions API to generate Persona content.
4. WHERE Claude is configured as the LLM_Provider, THE LLM_Client SHALL use the Anthropic Messages API to generate Persona content.
5. THE PersonaGenerator_API SHALL expose the same request and response schema regardless of which LLM_Provider is configured.
6. THE LLM_Client SHALL apply a request timeout of 30 seconds to all LLM_Provider calls, and IF the timeout is exceeded, THEN THE PersonaGenerator_API SHALL return HTTP 504 with a descriptive error message.

---

### Requirement 8: PersonaBuilder Frontend Component

**User Story:** As a user, I want a browser-based form to enter my age, years ahead, goals, and emotional baseline, so that I can generate my future self persona without leaving the application.

#### Acceptance Criteria

1. THE PersonaBuilder SHALL provide a numeric slider control for `current_age` with a range of 18 to 100.
2. THE PersonaBuilder SHALL provide a numeric slider control for `years_ahead` with a range of 1 to 50.
3. THE PersonaBuilder SHALL provide a goal tag input that allows the user to add between 1 and 10 goal strings, each no longer than 200 characters.
4. THE PersonaBuilder SHALL provide a selection control for `emotional_baseline` offering exactly the options `"anxious"`, `"hopeful"`, and `"neutral"`.
5. WHEN the user submits the form, THE PersonaBuilder SHALL send a POST request to `/persona/generate` with the UserProfile as a JSON body.
6. WHEN the PersonaGenerator_API returns HTTP 201, THE PersonaBuilder SHALL render the `summary`, `tone`, and `key_life_events` fields of the returned Persona to the user.
7. WHEN the PersonaGenerator_API returns an error response, THE PersonaBuilder SHALL display a human-readable error message corresponding to the HTTP status code.
8. WHILE a generation request is in progress, THE PersonaBuilder SHALL disable the submit control and display a loading indicator.
9. THE PersonaBuilder SHALL display the `persona_id` of a successfully generated Persona so that the user can reference it for downstream use.

---

### Requirement 9: Downstream Compatibility with Scenario Engine and Dialogue Generator

**User Story:** As a developer, I want the Persona output to be directly consumable by the Scenario Engine and Dialogue Generator, so that the modules integrate without data transformation.

#### Acceptance Criteria

1. THE PersonaGenerator_API SHALL return `persona_id` values as non-empty UUID strings with no whitespace.
2. WHEN a Persona is stored in the Persona_Store, THE PersonaGenerator_API SHALL return the identical field values in all subsequent GET responses for that record.
3. THE PersonaGenerator_API SHALL NOT transform, truncate, or re-encode any Persona field value between storage and retrieval.
4. THE PersonaGenerator_API SHALL include `tone` as a top-level string field in the Persona JSON so that the Dialogue_Generator can apply it directly to voice synthesis instructions.
5. THE PersonaGenerator_API SHALL include `key_life_events` as a top-level list field in the Persona JSON so that the Scenario_Engine can iterate over it directly without parsing nested structures.
6. THE PersonaGenerator_API SHALL include `key_message` as a top-level string field so that the Dialogue_Generator can use it to anchor the closing line of the generated script.
7. THE PersonaGenerator_API SHALL include `scenario_type` as a top-level string field in the Persona JSON so that the Scenario_Engine can use it to select the correct scenario category.

---

### Requirement 10: API Error Handling and Observability

**User Story:** As a developer, I want all API errors to be logged and return structured responses, so that issues can be diagnosed quickly in production.

#### Acceptance Criteria

1. WHEN any request to the PersonaGenerator_API results in an HTTP 4xx or 5xx response, THE PersonaGenerator_API SHALL log the error with a timestamp, request path, HTTP method, and error detail.
2. WHEN an unhandled exception occurs during request processing, THE PersonaGenerator_API SHALL return HTTP 500 with a JSON body containing a `detail` field and SHALL NOT expose internal stack traces or raw LLM_Provider responses to the client.
3. THE PersonaGenerator_API SHALL return all error responses as JSON objects with at minimum a `detail` field describing the error.
4. WHEN the LLM_Client retries a failed LLM_Provider request, THE PersonaGenerator_API SHALL log each retry attempt with the attempt number, LLM_Provider name, and failure reason.

---

### Requirement 11: Output Quality Evaluation

**User Story:** As a developer, I want generated personas to be evaluated for narrative coherence and tone alignment before being returned, so that only high-quality outputs reach downstream modules.

#### Acceptance Criteria

1. WHEN a Persona is generated, THE PersonaGenerator_API SHALL verify that the `life_outcome` does not contradict the `scenario_type` (e.g., a `"regret"` persona must not describe a triumphant outcome).
2. WHEN a Persona is generated, THE PersonaGenerator_API SHALL verify that the `tone` value matches the expected tone for the given `scenario_type` as defined in Requirement 3.
3. WHEN a Persona is generated, THE PersonaGenerator_API SHALL verify that the `summary` word count is within the 100–500 word range before persisting.
4. WHEN any coherence check fails, THE PersonaGenerator_API SHALL retry the LLM_Provider request up to 2 additional times before returning HTTP 502.
5. THE PersonaGenerator_API SHALL log a `narrative_coherence` assessment (pass/fail) alongside each generated Persona record in the Persona_Store.

**User Story:** As a developer, I want all API errors to be logged and return structured responses, so that issues can be diagnosed quickly in production.

1. WHEN any request to the PersonaGenerator_API results in an HTTP 4xx or 5xx response, THE PersonaGenerator_API SHALL log the error with a timestamp, request path, HTTP method, and error detail.
